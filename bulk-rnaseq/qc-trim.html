原始品質管控與修剪

在分析之前先看清 reads 的健康狀態——garbage in, garbage out 是體學分析的鐵律。

Inspect read health before analysis — "garbage in, garbage out" is the iron law of omics.

🟢 Beginner🟡 Intermediate🔴 Advanced
開場故事

為什麼第一步就是看 reads？

想像你收到一批新到的 RNA-seq 資料：100 GB 的 FASTQ、20 個樣本、3 個月實驗成果。實驗夥伴信誓旦旦：「定序中心說品質沒問題！」於是你直接餵給 STAR、跑 DESeq2，得到 8000 個顯著 DEGs。論文也許能這樣寫，但結論可能是錯的。因為定序中心的「品質報告」只看儀器層級指標（chastity filter pass rate），不會告訴你接頭是否切乾淨、樣本間 GC 偏好是否一致、是否有 rRNA / mt-RNA 汙染。這些都需要分析師親自確認。2019 年 Bioinformatics 期刊有一則撤稿：一篇號稱「找到新癌症生物標記」的論文被指出是接頭序列沒 trim 乾淨，所謂的「新標記」其實是技術 artifact。QC 不只是流程儀式，而是科學完整性的守門員。

Imagine you receive a fresh RNA-seq batch: 100 GB of FASTQ, 20 samples, three months of wet-lab work. Your collaborator insists: "The sequencing facility said quality is fine!" So you feed it to STAR, run DESeq2, and get 8000 significant DEGs. The manuscript could write itself — except the conclusions might be wrong. In 2019 a Bioinformatics retraction was issued: a paper claiming a "novel cancer biomarker" turned out to have un-trimmed adapter sequences creating PCR-duplicate / mis-alignment artifacts — the "marker" was a technical artifact. QC is not ceremony; it is the gatekeeper of scientific integrity.

類比：把 reads 想像成從河裡撈起的塑膠瓶——理論上每個都是「容器」，但實際上有些漏底（低品質）、有些卡黏液（含接頭）、有些是兩個瓶子套在一起（雙端讀穿）。在你倒水進去之前，必須先把瓶子分類乾淨。
Analogy: Think of reads as plastic bottles fished from a river. Some have holes, some are coated in grime, some are stuck together. Sort them before pouring anything in.
QC 與 Trimming 的三大目標
① 檢視

用 FastQC / MultiQC 看「健檢報告」：品質沿讀長變化、GC、接頭比例、重複序列。

Use FastQC / MultiQC for a "health check": per-base quality, GC, adapter fraction, duplication.

② 清理

用 fastp / trim_galore 切接頭、修 3' 低品質、移除過短 reads。

Use fastp / trim_galore to clip adapters, trim 3' ends, drop too-short reads.

③ 記錄

把每樣本 pre/post-trim 統計記錄下來。日後 debugging 時最快的線索。

Record pre/post-trim stats per sample. Fastest debugging clue down the line.

修剪理論

三、Trimming 的兩種「東西」要切

① 接頭修剪

何時發生：insert 比 read 短時，read 讀穿到接頭。

When: insert shorter than read length — read sequences through into adapter.

  • 偵測：與已知接頭序列比對
  • 常見：Illumina TruSeq、Nextera
  • 影響：未 trim → 比對失敗或讀錯位置
  • Detection: align to known adapters
  • Common: Illumina TruSeq, Nextera
  • Impact: untrimmed → alignment fails or wrong locus
vs
② 品質修剪

何時發生：定序末段化學疲乏，3' Q 下滑。

When: chemistry exhaustion in late cycles — 3' Q drops.

  • 偵測：滑動視窗或 3' Q 門檻
  • 常用：Q20，window 4 bp
  • 影響：過嚴 → 丟訊號；不夠 → 錯配率升
  • Detection: sliding window or 3' Q threshold
  • Typical: Q20, 4-bp window
  • Impact: too strict → loses signal; too loose → mismatches

案例研究

四、真實案例：ENCODE 標準 QC 流程

ENCODE Project · 2003-present · 16,000+ samples

📂 ENCODE 怎麼做 QC？

真實資料 ENCODE Project NHGRI 自 2003 年運行至今，目前公開 > 16,000 個 RNA-seq 樣本。為讓資料可比，他們訂了嚴格 QC SOP：

  1. 原始 reads ≥ 25 M PE（標準）或 30 M（splice junction 分析）
  2. FastQC + MultiQC 報告必須隨資料發佈
  3. fastp Q20 + min length 25，UMI 樣本另用 umi_tools 抽取
  4. STAR mapping rate ≥ 80%（unique > 65%）
  5. rRNA ratio < 5%（SortMeRNA / Picard CollectRnaSeqMetrics）
  6. 3'/5' bias < 2.0（RIN ≥ 7 樣本應如此）

未通過 SOP 的樣本會被打 error_status 旗標，重做或註記。這套標準後來影響了 GTEx、TCGA、recount3 等大型 cohort。啟示：沿用 ENCODE 指標能讓你的資料更容易進入 meta-analysis。

Real Data ENCODE Project (NHGRI, since 2003) hosts >16,000 RNA-seq samples. To enforce cross-cohort comparability:

  1. Raw reads ≥ 25 M PE (or 30 M for splice junctions)
  2. FastQC + MultiQC reports co-published
  3. fastp Q20 + length 25; umi_tools first for UMI samples
  4. STAR mapping ≥ 80% (unique > 65%)
  5. rRNA < 5% (SortMeRNA / Picard)
  6. 3'/5' bias < 2.0 (for RIN ≥ 7 samples)

Failed samples get an error_status flag. This SOP shaped GTEx, TCGA, recount3. Takeaway: adopting ENCODE thresholds eases meta-analysis.

討論

六、思考題

💭 五道值得思考的問題

Q1. 拿到一批 single-end 50 bp 舊資料（2012）與 paired-end 150 bp 新資料（2024）混在一個 cohort 做 DE，你會用什麼策略？要把新資料 down-trim 到 50 bp 嗎？還是分批處理後 meta-analysis？

Q2. FastQC per-base sequence content 前 12 bp 偏離 25%/25%/25%/25%，你會 trim 嗎？答案是「不」——但為何這對 RNA-seq 正常、對 ChIP-seq 卻是嚴重警訊？（提示：random hexamer priming）

Q3. collaborator 堅持把所有 reads trim 到固定長度（如 75 bp），認為這樣可「公平比較」。如何說服他這是錯的？（提示：mapping rate、splice junction）

Q4. 樣本 mapping rate 只有 35%，FastQC 卻全綠。下一步？（提示：FastQ Screen、SortMeRNA、blast 典型 unmapped read）

Q5. 為何 ENCODE 把 rRNA < 5% 列為 SOP？想想 ribozero / polyA capture 化學差異——若樣本 rRNA 達 30%，是 library prep 失敗還是 RNA degradation？

Q1. A mixed cohort: old SE 50 bp (2012) + new PE 150 bp (2024) for joint DE. Strategy? Down-trim new to 50 bp, or batch-process and meta-analyse?

Q2. Per-base content deviates from 25%/25%/25%/25% in first 12 bp. Trim it? Answer: no — explain why this is normal for RNA-seq yet a warning for ChIP-seq. (Hint: random hexamer priming.)

Q3. A collaborator wants all reads trimmed to fixed length (75 bp) "for fair comparison." How to convince them it's wrong? (Hint: mapping rate, splice junction.)

Q4. Sample has 35% mapping rate but FastQC is all green. Next move? (Hint: FastQ Screen, SortMeRNA, BLAST a typical unmapped read.)

Q5. Why does ENCODE require rRNA < 5%? Consider ribozero vs polyA capture chemistry — if rRNA hits 30%, is it library prep failure or RNA degradation?

分級學習

七、依程度學習

剛接觸 RNA-seq？掌握三件事：

  1. 跑一次 fastp 預設參數（-q 20 -l 36）
  2. 用 MultiQC 整合所有 fastp 報告
  3. 檢查 duplication 與 GC% 是否有極端 outlier

建議練習資料：Bioconductor airway 或 pasilla，乾淨且公開。

New to RNA-seq? Three priorities:

  1. Run fastp once with defaults (-q 20 -l 36)
  2. MultiQC to merge all fastp reports
  3. Look for outliers in duplication and GC%

Practice data: Bioconductor airway or pasilla.

熟悉基本流程？進階：

  1. RNA-seq 特有 QC：Picard CollectRnaSeqMetrics 看 5'/3' bias、coverage 均勻度
  2. 跨樣本 QC：PCA 觀察樣本群聚；若 case/ctrl 沒分開卻沿 batch 軸分開 = batch effect
  3. 汙染檢測：FastQ Screen / Kraken2 同時對多物種比對
  4. UMI 流程：對 SC 或低輸入樣本是 counts 關鍵

Comfortable with basics? Level up:

  1. RNA-seq-specific QC: Picard CollectRnaSeqMetrics — 5'/3' bias, coverage uniformity
  2. Cross-sample QC: PCA — if case/ctrl don't separate but batch does, batch effect
  3. Contamination: FastQ Screen / Kraken2 vs multiple genomes
  4. UMI workflows: critical for SC and low-input

進階研究員 / 流程開發者？

  1. Error-mode 行為：BBDuk vs fastp 在 reverse-complement adapter detection 精度差異
  2. Q-score recalibration：DNA-seq 用 GATK BQSR；SmartSeq2 等 PCR-heavy 也可能受益
  3. PolyG/polyX 邊界：保留 mRNA 真實 polyA 尾 vs 去除 NovaSeq polyG（fastp --poly_g_min_len 細節）
  4. 批次效應與 trim 異質性：不同樣本若用不同 trim 參數會引入隱性 batch effect
  5. nf-core / Snakemake 整合：撰寫可重現 QC 模組

Advanced researcher / pipeline developer:

  1. Error-mode behavior: BBDuk vs fastp reverse-complement adapter detection differences
  2. Q-score recalibration: GATK BQSR for DNA-seq; SmartSeq2 PCR-heavy may benefit
  3. PolyG/polyX boundary: preserve real polyA vs remove NovaSeq polyG (fastp --poly_g_min_len)
  4. Batch effects from trim heterogeneity: different trim params introduce hidden batch effects
  5. nf-core / Snakemake integration — reproducible QC modules

八、常見問答

技術可行——STAR/Salmon 對輕微未 trim 接頭有容忍度。但實務上：

  1. 未 trim 略降 mapping rate（2-5%）
  2. FastQC adapter content 紅色殘留
  3. 對 splice junction detection 干擾

建議仍跑一次 trim（fastp 預設），時間成本 < 5 分鐘/樣本，效益值得。

Technically yes — but practically:

  1. Drops mapping rate 2–5%
  2. FastQC adapter content stays red
  3. Interferes with splice-junction detection

Recommend: still run fastp defaults (<5 min/sample).

常見原因：(1) 非標準接頭（NEBNext / Lexogen 用 Illumina 不同接頭），需顯式指定；(2) FastQC 把高重複序列誤判為接頭（如特定 mtRNA 區段）；(3) NovaSeq polyG 假象，加 fastp --trim_poly_g。

Common causes: (1) non-standard adapters (NEBNext/Lexogen use different sequences); (2) FastQC mistakes repeats for adapters; (3) NovaSeq polyG artifacts — add fastp --trim_poly_g.

正常——R2 在 R1 後才開始定序，3' 端品質下滑更快。但必須同步 trim R1/R2：若一邊 trim 到 50 bp、另一邊 150 bp，就不再 paired。fastp/trim_galore 預設會自動保持 pair 關係。

Normal — R2 sequences later, 3' degrades faster. But you must trim R1/R2 in sync: if one is 50 bp and the other 150 bp, they're no longer paired. fastp/trim_galore preserve pair relationships by default.

不一定：(a) rRNA 殘留（GC ~60%）與 mRNA peak（41%）形成雙峰，ribozero 不徹底時常見；(b) 細菌/病毒汙染（GC 變化大）；(c) 正常組織（如肝）高表達基因 GC bias 強。建議先看 rRNA 比例，再做 Kraken2/FastQ Screen 確認。

Not always: (a) rRNA carry-over (GC ~60%) + mRNA peak (41%) = bimodal — common with incomplete ribozero; (b) bacterial/viral contamination; (c) some tissues (liver) have strong GC bias. Check rRNA first, then Kraken2/FastQ Screen.

業界經驗：mapping rate < 50%、rRNA > 30%、5'/3' bias > 4、trim 後 usable reads < 10 M、PCA 嚴重 outlier 之一發生即考慮重做或剔除。

Industry rules: any of — mapping < 50%, rRNA > 30%, 5'/3' bias > 4, usable reads < 10 M, severe PCA outlier — consider resequencing or excluding.
